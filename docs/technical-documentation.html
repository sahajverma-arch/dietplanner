App reads/writes execute as the signed-in user via the cookie-bound Supabase client, so Row Level Security is the source of authorization. The service-role key exists only server-side for the sheet sync and the one-off food seeding script.

4. Counselling Engine (src/lib/counselling/)

The first counselling is the LeanR Premium Final New Client First Counselling — Q1–Q105 in 12 sections (goal & motivation · body history · medical safety · digestion · actual food day · preferences & feasibility · training/protein/recovery · routine & behaviour · lifestyle · success/dropout/coaching · dietitian assessment · client discussion), 55–65 minutes, conversational option-select. It is entirely schema-as-data in questions.ts; ClinicalCounsellingForm.tsx renders whatever the schema says, so rewording or adding a question never touches UI code.

  • Question types: text, textarea, number, time, date, single, multi (with optional max — "select up to N", click order doubles as rank), 1–10 sliders.
  • Conditional visibility: showIf(answers) per question — numeric targets appear per selected Q5 target, safety follow-ups on Q21 symptoms, day-of-week rules under Q38, the hormonal section only where relevant, training detail only when training.
  • Meal timeline (Q28): selecting a meal occasion generates its q28_<occasion>_{time,food,prep,source,extras} fields — a structured 24-hour recall.
  • Mandatory gate: missingRequired() (visibility-aware) blocks generation in the UI and again in the API.
  • assessment.ts: redFlags() — urgent Q21 symptoms, severe allergy, disordered-eating risk (Q60), pregnancy, renal/bariatric, RED-S under-fuelling, each marked escalate vs caution; audit() — 100-point quality rubric across 11 categories, computed live; toIntake() — legacy projection keeping the clients table and follow-ups unchanged; aiProfile() — the 14-block clinical profile the AI analyses (including dietitian_hypothesis = Section 11 and red_flags).

5. AI Pipeline (src/lib/nim.ts)

5.1 Independent clinical review (before every Week-1 plan)

Per the LeanR Premium spec, the dietitian's Section 11 strategy is a professional hypothesis. aiClinicalReview() sends the complete profile and receives a Zod-validated decision object:

{ "decision": "Support Dietitian Strategy" | "… With Minor Modification" |
              "Significantly Modify Dietitian Strategy" |
              "Use AI-Led Alternative Clinical Nutrition Strategy" |
              "Pause Final Diet Generation and Request Clinical Review or Missing Information",
  "reasoning": string, "strategy_adjustments": string[],
  "missing_information": string[], "safety_concerns": string[] }

A Pause aborts generation (HTTP 422) before the client row is created, so the counselling remains an editable draft. Any other decision is woven into the plan prompts and stored in diet_plans.ai_review. A failed review call degrades to generation-without-review (never blocks the dietitian).

5.2 Plan generation

The shared NIM endpoints cap completions around 2k tokens, so the 7-day plan is generated in two calls (overview + days 1–4, then days 5–7 seeing days 1–4 for variety) and merged. Each call is hardened: response_format: json_object with graceful degradation, 120 s per-call timeout, automatic fallback model on transient failures, Zod validation with corrective retries. Truncation is handled explicitly — unparseable output and "valid JSON but too few days" (constrained decoding closing the object at the completion cap) both trigger a brevity-focused retry instead of echoing the broken response; if the primary model still cannot produce a valid plan after 3 attempts, the whole conversation is rerun once on the fallback model before failing.

Revision rounds: when the dietitian requests changes on a draft (§7), generateDietPlan() receives the current draft plus the written instructions. The prompt marks the instructions as mandatory (second only to allergies and medical safety) and directs the model to revise the existing draft — keeping everything not mentioned — rather than start over. starts_on is passed through so weekday-specific rules keep their original Day-1 mapping across revision rounds.

The system prompt encodes the Final Diet Generation Principles: never plan from the goal / dietitian strategy / calorie maths / weight alone; obey red flags and doctor instructions; follow the review decision; smallest number of high-impact changes matched to readiness; start from the actual food day; protect favourites, non-negotiables and cultural foods; avoid unnecessary restriction; avoid the client's dropout pattern; fuel muscle, training and recovery; design for the client's actual life.

5.3 Deterministic enforcement (not left to the model)

  • Forbidden foods: allergens + intolerances + dislikes are listed in the prompt and scanned in the returned meals; violations are named back for correction; surviving dislikes are stripped; a surviving allergen aborts the plan.
  • Day-of-week rules (q38a–c): "Day N" maps to real weekdays via starts_on (Day 1 = tomorrow, fixed at generation so revision rounds and the approval-time PDF agree); restricted weekdays are checked term-by-term (non-veg, eggs, onion/garlic, all animal products) and violations stripped as a last resort.
  • Every revision round re-runs all of the above — a dietitian instruction can never reintroduce an allergen, diet-type violation or day-rule breach, and the revised plan is re-grounded against the foods table.

5.4 Nutrition grounding (src/lib/nutrition.ts)

All item names go to Postgres in one RPC (match_foods_batch: pg_trgm fuzzy matching, blended scoring with a preference for INDB), quantities like "2 rotis" / "1 katori" / "150 g" resolve to grams via household serving weights, and a meal's macros are replaced only when every item matched and resolved. Non-fatal by design: an empty foods table just keeps model estimates.

6. API Reference

POST /api/generate-plan (auth: session cookie · Node · maxDuration 300)

BodyBehaviour
{"type":"first","form": ClinicalIntake, "appointmentId"?}Server-side mandatory gate → AI clinical review (Pause → 422 {error, paused, aiReview}, nothing persisted) → create client → generate → ground → insert plan as draft (+ai_review, starts_on) → mark appointment Completed → clear draft. Returns {clientId, planId, week, status:"draft"}. No PDF yet.
{"type":"followup","clientId","followup"}Inserts the follow-up, updates client weight, generates week latest+1 as a draft using the LeanR weekly loop and last week's menu.
{"type":"regenerate","clientId"}Retries from stored intake as a draft; re-runs the AI review when retrying a Week-1 plan.
{"type":"revise","planId","instructions"}Human-in-the-loop round on a draft: rebuilds the original context (intake, AI review, previous week's plan, follow-up data), regenerates with the dietitian's instructions marked mandatory, re-runs enforcement + grounding, updates the same row and appends {instructions, at} to revisions. Draft only — an approved plan returns 409.
{"type":"approve","planId"}Renders the PDF from the reviewed plan (dates from starts_on), uploads to storage, sets status = 'final' + pdf_path. No model calls (~2 s). Draft only — 409 otherwise.

Errors: 400 invalid body/gate · 401 unauthenticated · 404 unknown client/plan · 409 plan already approved · 422 review pause or unparseable stored draft · 502 {error, clientId?} on generation/PDF/storage failure (the UI routes to the client page where retry is offered).

Long-request behaviour: generation is a synchronous request (1–3 min typically; longer when the primary model is congested and per-call timeouts cascade). If the browser connection drops mid-request, the server still finishes and saves the draft — refresh the client page instead of resubmitting the counselling form, which would create a duplicate client.

POST /api/sync-counselling

Guarded by CRON_SECRET (x-cron-secret header or ?secret=). Reads the Raw_Counselling sheet (Google service account preferred, CSV-export fallback), upserts counselling_appointments on (client_code, emp_code, scheduled_on) touching only sheet columns — workflow status survives re-syncs. Triggered daily by QStash and manually from /admin.

7. Human-in-the-Loop Plan Review (PlanReviewControls)

Every generated plan — first counselling, follow-up or regenerate — is stored as a draft preview and shown on the client page under an amber "Draft preview — needs your review" panel. The dietitian inspects the full plan (meals, macros, repetition, timing) before anything reaches the client. The panel offers:

  • Free-text change instructions — e.g. "too much paneer, vary the protein sources; keep protein near 90 g; Day 3 dinner lighter" — sent with type:"revise". The AI revises the existing draft (not a fresh plan), the same safety enforcement and grounding re-run, and the round is appended to the plan's revisions history, displayed as "Changes already applied". Unlimited rounds.
  • Approve & create PDF — type:"approve" renders and stores the final PDF and flips the row to final; the download button appears in plan history.

Rationale: generated diets occasionally repeat foods across the week or overshoot protein; the review checkpoint puts the dietitian's judgement between the model and the client, with their corrections applied by the AI in ~1–2 minutes instead of manual re-editing. Plan history marks unapproved rows "Draft — in review"; drafts have no PDF and never reach a client. The counselling form's autosaved draft is still cleared at generation time (the counselling itself succeeded); plan-level iteration happens entirely on the client page.

8. PDF Rendering (src/lib/pdf.tsx)

  • Rendered at approval time from the reviewed plan; day dates come from the stored starts_on, so what the dietitian approved is exactly what the PDF shows.
  • Premium macro-report layout: black header band with client name + LEANR logo, macro target cards, per-day tables (Time | Meal | Foods | Calories | Protein | Carbs | Fat | Cal%), highlighted Daily Total rows, Weekly Summary with P%/C%/F%, guidelines, foods to avoid, real dates per day, and a two-line disclaimer footer on every page.
  • Daily totals, Cal% and the weekly summary are computed from meal data, never trusted from the model.
  • Preview without the AI: npx tsx scripts/render-sample-pdf.tsx out.pdf.

9. Autosave Design

ClinicalCounsellingForm debounces every change (1.2 s) into an upsert on form_drafts keyed by (dietitian_id, kind), storing answers + the source appointment id, with a live Saving/Saved/Failed indicator. The draft restores server-side on load (only for the same appointment) and is deleted exclusively after successful generation — a failed generation, review Pause, or closed tab never loses call notes.

11. Setup & Deployment

  1. Supabase: run migrations 0001 → 0007 in order in the SQL Editor (tables, RLS, trigger, bucket/policies, foods matching, Google-auth constraints, appointments, ai_review, draft-review columns).
  2. Food data: put the service-role key in .env.local, run npm run seed:foods (~7,900 rows; idempotent upsert).
  3. Google sign-in: create an OAuth client in Google Cloud Console, add Supabase's redirect URI, enable the Google provider in Supabase Auth.
  4. Local: npm install → fill .env.local from .env.example → npm run dev.
  5. Vercel: import the repo, set env vars, deploy; add the domain to Supabase Auth URL configuration; create the QStash daily schedule (POST /api/sync-counselling, cron 30 0 * * *, secret header).
  6. QA: npx tsx scripts/test-clinical-counselling.mts (flags → audit → review → plan → adherence checks), npx tsx scripts/e2e-test-plan.ts (two clients → PDFs in test-output/) and npx tsx scripts/test-revision-loop.ts (draft → written instructions → revised draft, with protein/variety checks).
